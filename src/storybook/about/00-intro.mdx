  # Diamond Design System ❖

  <p>
    The design system for Diamond Light Source, providing a shared foundation for
    building clear, consistent, and reliable scientific interfaces.
  </p>

  <p>
    It evolves SciReactUI into a consistent, semantic, and scalable system for data
    acquisition, analysis, and operational tools across Diamond.
  </p>

  <p>
    Built on structure and precision, the system supports clear, coherent, and
    predictable interfaces across scientific workflows and applications.
  </p>

  ## What it helps us do

  <ul>
    <li>Create clearer, more predictable scientific interfaces.</li>
    <li>Bring consistency across tools and beamlines.</li>
    <li>Reduce cognitive load in complex workflows.</li>
    <li>Speed up development with reusable components.</li>
    <li>Ensure new tools start coherent and stay aligned.</li>
  </ul>

  ## Why we need a design system

  <p>
    Diamond operates a growing ecosystem of tools across beamlines, data
    acquisition, analysis, and scientific support. Many of these tools have
    evolved independently, with different patterns, behaviours, and visual
    languages.
  </p>

  <p>
    Diamond II creates a clear moment to modernise these interfaces around
    clarity, safety, efficiency, and consistency.
  </p>

  <ul>
    <li>
      <strong>User-centred interfaces</strong>: interfaces are built around real
      user needs, reducing friction, errors, and workarounds.
    </li>
    <li>
      <strong>Unify systems</strong>: shared patterns reduce inconsistency across
      tools and workflows.
    </li>
    <li>
      <strong>Modernise for Diamond II</strong>: new beamlines and workflows give
      us a clear opportunity to redesign interfaces for clarity, safety, and
      efficiency.
    </li>
    <li>
      <strong>Reduce cognitive load</strong>: familiar, intuitive interfaces
      reduce training time and mental effort when moving between systems.
    </li>
    <li>
      <strong>Improve collaboration</strong>: shared standards create a common
      language between developers, scientists, designers, and stakeholders.
    </li>
    <li>
      <strong>Accelerate development</strong>: reusable components and patterns
      reduce duplication and keep design and engineering aligned.
    </li>
    <li>
      <strong>Future-proof our ecosystem</strong>: every new tool, feature, or
      beamline interface can start from a coherent foundation.
    </li>
  </ul>

  ## What makes scientific interfaces different

  <ul>
    <li>
      <strong>Time is scarce</strong>: beamline access is limited, and
      interruptions are costly.
    </li>
    <li>
      <strong>Experiments are stateful</strong>: the interface must make system
      state, progress, and risk clear.
    </li>
    <li>
      <strong>Users may be visiting</strong>: scientists often need to become
      productive quickly, under pressure.
    </li>
    <li>
      <strong>High-density data is normal</strong>: tables, logs, dashboards,
      plots, and configuration views are core to the experience.
    </li>
    <li>
      <strong>Actions can be high consequence</strong>: controls, destructive
      actions, and state changes need clear signifiers and recovery paths.
    </li>
  </ul>
