## Using the theme

Diamond Design System uses semantic design tokens mapped into MUI’s theme
system.

CSS variables remain the source of truth. The MUI theme adapts those values into
component APIs, palette roles, typography, spacing, elevation, and interaction
states.

Prefer theme roles and semantic tokens over hardcoded values.

Using semantic roles keeps interfaces consistent across products, supports light
and dark themes, and allows the system to evolve without rewriting component
styles.

## Using components

Build with existing components and patterns before creating new ones.

Components should keep interfaces predictable, readable, and consistent.

## Colour usage

Use colour semantically. Separate structure, meaning, and brand.

DiamondDS supports:

- action intents: primary, secondary
- status intents: success, warning, error, info

Intent colours communicate hierarchy, operational meaning, and state through component APIs such as `color="primary"` or `color="error"`.
Their meaning depends on context and workflow.

Brand colour is intentionally separate. Brand communicates Diamond identity rather than behaviour or status.

## Borders

Borders define separation, grouping and affordance. They should support structure without creating visual noise.

Use border roles instead of raw greys:

<table className="ds-table">
  <thead>
    <tr>
      <th>Role</th>
      <th>MUI mapping</th>
      <th>Purpose</th>
      <th>Typical use</th>
    </tr>
  </thead>
  <tbody>
    <tr>
      <td>
        <code>--ds-border-subtle</code>
      </td>
      <td>
        <code>divider</code>
        <span> / </span>
        <br />
        <code>border.subtle</code>
      </td>
      <td>Default quiet border</td>
      <td>Dividers, table separators, low-emphasis boundaries</td>
    </tr>
    <tr>
      <td>
        <code>--ds-border-emphasis</code>
      </td>
      <td>
        <code>border.emphasis</code>
      </td>
      <td>Elevated interaction/emphasis</td>
      <td>Inputs, highlighted cards, panels, tables, important grouping</td>
    </tr>
    <tr>
      <td>
        <code>--ds-border-strong</code>
      </td>
      <td>
        <code>border.strong</code>
      </td>
      <td>Maximum border contrast</td>
      <td>Hover states, active regions, selected-neutral</td>
    </tr>
  </tbody>
</table>

MUI’s native <code>divider</code> role and DiamondDS <code>border.subtle</code> use the same value.
Use <code>divider</code> where a MUI component expects it, and <code>border.subtle</code> when writing DiamondDS border styles directly.

<div className="ds-dos">
  <div className="ds-card">
    <h3>✅ Do</h3>
    <ul>
      <li>Use <code>border.subtle</code> for quiet dividers, separators, table boundaries, and disabled outlines.</li>
      <li>Use <code>border.emphasis</code> for interactive component boundaries and elements that require clearer affordance.</li>
      <li>Use <code>border.strong</code> for elevated interaction states such as hover, active, or selected elements.</li>
      <li>Use intent border colours only for meaningful states such as error or selected.</li>
      <li>Let disabled and error borders override hover and focus styling.</li>
    </ul>
  </div>

  <div className="ds-card">
    <h3>❌ Don’t</h3>
    <ul>
      <li>Don’t use raw grey values for component borders.</li>
      <li>Don’t use borders as the only way to communicate status.</li>
      <li>Don’t add heavy borders to compensate for weak spacing or grouping.</li>
      <li>Don’t mix several border strengths in the same small area.</li>
      <li>Don’t invent one-off border colours outside the theme.</li>
    </ul>
  </div>
</div>

## Button hierarchy and pairing

<div className="ds-note">
  Button hierarchy should make the next safe action obvious. Avoid placing
  multiple high-emphasis actions next to each other unless they genuinely have
  equal priority.
</div>

### Default pairing

Use one contained button for the main action, then reduce emphasis for nearby
supporting actions.

### Recommended patterns

<table className="ds-table">
  <thead>
    <tr>
      <th>Pattern</th>
      <th>Use when</th>
      <th>Recommended pairing</th>
    </tr>
  </thead>
  <tbody>
    <tr>
      <td>Main action + cancel</td>
      <td>The user needs a clear way to proceed or back out</td>
      <td>
        <code>primary contained</code> + <code>text</code>
      </td>
    </tr>
    <tr>
      <td>Main action + supporting action</td>
      <td>The second action helps prepare or configure the main action</td>
      <td>
        <code>primary contained</code> + <code>outlined</code>
      </td>
    </tr>
    <tr>
      <td>Main action + more actions</td>
      <td>Additional actions are available but should not compete</td>
      <td>
        <code>primary contained</code> + menu button
      </td>
    </tr>
    <tr>
      <td>Destructive confirmation</td>
      <td>The user is confirming a destructive or irreversible action</td>
      <td>
        <code>error contained</code> + neutral <code>text</code> or{" "}
        <code>outlined</code>
      </td>
    </tr>
    <tr>
      <td>Promoted secondary workflow</td>
      <td>A secondary action is intentionally more useful in context</td>
      <td>
        Lower-emphasis primary action + <code>secondary contained</code>
      </td>
    </tr>
  </tbody>
</table>

<div className="ds-dos">
  <div className="ds-card">
    <h3>✅ Do</h3>
    <ul>
      <li>Use one high-emphasis action in a local action group.</li>

      <li>
        Use text buttons for cancel, dismiss, and low-risk exits.
      </li>

      <li>
        Use menu buttons for overflow actions that should not compete visually.
      </li>

      <li>
        Use destructive emphasis only when the action is genuinely destructive
        or hazardous.
      </li>
    </ul>

  </div>

  <div className="ds-card">
    <h3>❌ Don’t</h3>
    <ul>
      <li>
        Don’t place <code>primary contained</code> and
        <code>secondary contained</code> next to each other by default.
      </li>

      <li>
        Don’t use multiple contained buttons to solve unclear priority.
      </li>

      <li>
        Don’t make cancel actions visually compete with the main action.
      </li>

      <li>
        Don’t promote secondary actions just for visual variety.
      </li>
    </ul>

  </div>
</div>

## Feedback and status

<div className="ds-note">
  Interfaces should continuously communicate system state, progress, success,
  warnings, and problems. Clear feedback helps users understand what is
  happening, what requires attention, and whether it is safe to continue.
</div>

### Communicate state clearly

Provide feedback for important actions, transitions, and system events.

Users should understand:

<ul>
  <li>what is happening</li>
  <li>whether an action succeeded</li>
  <li>whether attention is required</li>
  <li>whether work is still in progress</li>
  <li>what they can safely do next</li>
</ul>

Avoid relying on colour alone. Combine colour with labels, icons, text, or
layout changes where appropriate.

### Progress and transitional states

Use informational feedback for active or changing system states.

Examples include:

<ul>
  <li>scanning in progress</li>
  <li>moving motors</li>
  <li>connecting to device</li>
  <li>processing data</li>
  <li>queue paused</li>
</ul>

Prefer persistent inline or contextual feedback over temporary notifications for
long-running operations.

---

### Success feedback

Use success feedback sparingly.

Not every successful action requires a notification.

Prefer lightweight confirmation for routine actions, especially in repetitive
scientific workflows.

<table className="ds-table">
  <thead>
    <tr>
      <th>✅ Prefer</th>
      <th>❌ Avoid</th>
    </tr>
  </thead>
  <tbody>
    <tr>
      <td>Quiet inline confirmation after saving</td>
      <td>Large celebratory success states for routine operations</td>
    </tr>
    <tr>
      <td>Small status update in a queue or table row</td>
      <td>Stacking repeated success toasts during repetitive workflows</td>
    </tr>
    <tr>
      <td>Temporary confirmation for destructive actions</td>
      <td>Interrupting workflow with unnecessary success messaging</td>
    </tr>
  </tbody>
</table>

## Errors and recovery

<div className="ds-note">
  Error handling should help users recover safely and confidently. Focus on
  clarity, next steps, and preventing further mistakes.
</div>

### Make errors actionable

Good error messages explain:

<ul>
  <li>what happened</li>
  <li>what caused the issue, if known</li>
  <li>what the user can do next</li>
</ul>

Avoid vague or technical-only messaging where possible.

    </tr>
  </thead>

### Prefer recovery over dead ends

Where possible, help users continue safely.

Provide:

<ul>
  <li>retry actions</li>
  <li>alternative paths</li>
  <li>clear recovery guidance</li>
  <li>safe defaults</li>
</ul>

Avoid trapping users in blocked states without explanation.

### Safety and operational awareness

In scientific and operational environments, errors may affect hardware,
beamtime, experiments, or data integrity.

Use stronger emphasis when actions may:

<ul>
  <li>interrupt experiments</li>
  <li>move hardware</li>
  <li>discard data</li>
  <li>affect safety</li>
  <li>impact other users or systems</li>
</ul>

Destructive or irreversible actions should require clear confirmation and should
never compete visually with safe actions.
