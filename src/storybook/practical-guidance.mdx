# Practical guidance

Diamond Design System uses MUI with Diamond semantic palette roles layered into the theme.

The MUI theme is the source of truth for colour, surfaces, borders and component states. Light and dark mode are handled through MUI colour schemes.

## Using the theme

Prefer semantic tokens, existing component variants, and reusable patterns.

Avoid hardcoded values, one-off styling, and inconsistent interaction patterns.

## Using components

Build with existing components and patterns before creating new ones.

Components should keep interfaces predictable, readable, and consistent.

      <li>Don’t create multiple patterns for the same interaction.</li>
      <li>Don’t override the theme without a reusable reason.</li>
    </ul>

## Colour usage

Use colour semantically. Separate structure, meaning, and brand.

DiamondDS supports:

- action intents: primary, secondary
- status intents: success, warning, error, info

Intent colours communicate hierarchy, operational meaning, and state through component APIs such as `color="primary"` or `color="error"`.
Their meaning depends on context and workflow.

Brand colour is intentionally separate. Brand communicates Diamond identity rather than behaviour or status.

## Surfaces and elevation

Surfaces create structure and hierarchy. They should help users understand
where content belongs without making dense scientific interfaces feel noisy.

Prefer layered surfaces and borders over heavy shadows.

## Borders

Borders define separation, grouping and affordance. They should support structure without creating visual noise.

Use border roles instead of raw greys:

<table className="ds-table">
  <thead>
    <tr>
      <th>MUI mapping</th>
      <th>Purpose</th>
      <th>Typical use</th>
    </tr>
  </thead>
  <tbody>
    <tr>
      <td>
        <code>divider</code>
        <span> / </span>
        <br />
        <code>border.subtle</code>
      </td>
      <td>Quiet separation</td>
      <td>Dividers, table row separators, low-emphasis boundaries</td>
    </tr>
    <tr>
      <td>
        <code>border.base</code>
      </td>
      <td>Default boundary</td>
      <td>Inputs, cards, panels, table cells</td>
    </tr>
    <tr>
      <td>
        <code>border.emphasis</code>
      </td>
      <td>Stronger boundary</td>
      <td>Hover states, active regions, important grouping</td>
    </tr>
  </tbody>
</table>

MUI’s native <code>divider</code> role and DiamondDS <code>border.subtle</code> use the same value.
Use <code>divider</code> where a MUI component expects it, and <code>border.subtle</code> when writing DiamondDS border styles directly.

<div className="ds-dos">
  <div className="ds-card">
    <h3>Do</h3>
    <ul>
      <li>Use <code>border.base</code> for standard component boundaries.</li>
      <li>Use <code>border.subtle</code> for quiet dividers and disabled outlines.</li>
      <li>Use <code>border.emphasis</code> when a boundary needs stronger affordance.</li>
      <li>Use intent border colours only for meaningful states such as error or selected.</li>
      <li>Let disabled and error borders override hover and focus styling.</li>
    </ul>
  </div>

  <div className="ds-card">
    <h3>Don’t</h3>
    <ul>
      <li>Don’t use raw grey values for component borders.</li>
      <li>Don’t use borders as the only way to communicate status.</li>
      <li>Don’t add heavy borders to compensate for weak spacing or grouping.</li>
      <li>Don’t mix several border strengths in the same small area.</li>
      <li>Don’t invent one-off border colours outside the theme.</li>
    </ul>
  </div>
</div>

## Typography usage

Typography should support readability, scanning, and technical clarity.

  <thead>

      <th>Typical usage</th>
    </tr>
  </thead>
  <tbody>
    <tr>
      <td>Inter</td>
      <td>Default interface typeface</td>
      <td>UI text, controls, tables, navigation</td>
    </tr>
    <tr>
      <td>Outfit</td>
      <td>Brand and high-level headings</td>
      <td>Product headings, landing areas</td>
    </tr>
    <tr>
      <td>IBM Plex Mono</td>
      <td>Technical and aligned content</td>
      <td>IDs, timestamps, code, numeric values</td>
    </tr>
  </tbody>
</table>

<div className="ds-dos">
  <div className="ds-card">
    <h3>Do</h3>
    <ul>
      <li>Keep hierarchy consistent across products.</li>
      <li>Prefer short, clear labels.</li>
      <li>Make numbers and technical values easy to scan.</li>
      <li>Use mono type where alignment matters.</li>
    </ul>
  </div>

  <div className="ds-card">
    <h3>Don’t</h3>
    <ul>
      <li>Don’t use brand typography for dense operational content.</li>
      <li>Don’t rely on decorative styling for hierarchy.</li>
      <li>Don’t hide important information in faint text.</li>
      <li>Don’t introduce one-off font sizes.</li>
    </ul>
  </div>
</div>

## General principles

<ul>
  <li>Prefer theme palette roles and component props over raw values.</li>
  <li>Reuse patterns before creating new ones.</li>
  <li>Keep interactions predictable.</li>
  <li>Design for clarity and operational confidence.</li>
  <li>Similar things should look and behave similarly.</li>
</ul>
