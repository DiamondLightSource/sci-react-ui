# Core principles

<p>
  These principles guide how we design and build interfaces across the Diamond
  Design System.
</p>

</div>

<div className="ds-principles">

<div className="ds-card ds-card-alt">
  <h3>Clarity</h3>
  <p>
    Interfaces should be easy to understand at a glance. Users should be able to
    see system state and next actions without interpreting multiple signals.
  </p>
</div>

<div className="ds-card ds-card-alt">
  <h3>Consistency</h3>
  <p>
    Similar elements should look and behave the same across the system, reducing
    the need to relearn interactions.
  </p>
</div>

<div className="ds-card ds-card-alt">
  <h3>Accessibility</h3>
  <p>
    Accessibility is a baseline, not an enhancement. Interfaces must remain
    usable across a wide range of abilities and conditions.
  </p>
</div>

<div className="ds-card ds-card-alt">
  <h3>Predictable behaviour</h3>
  <p>
    Interactions should behave as users expect. Reliability is critical in
    stateful and high-consequence workflows.
  </p>
</div>

<div className="ds-card ds-card-alt">
  <h3>Theme-driven</h3>
  <p>
    Visual decisions are defined through the DiamondDS theme, so colours,
    surfaces and component states remain consistent across light and dark mode.
  </p>
</div>

<div className="ds-card ds-card-alt">
  <h3>Semantic first</h3>
  <p>
    Use roles and meaning, not raw values. For example, use
    <code> theme.palette.surface.subtle</code> or component props instead of
    hardcoded colours.
  </p>
</div>

<div className="ds-card ds-card-alt">
  <h3>Separation of concerns</h3>
  <p>
    Palette roles, components and implementation are distinct layers, allowing
    the system to scale safely.
  </p>
</div>

</div>

## Do and don’t

<div className="ds-dos">
  <div className="ds-card">
    <h3>Do</h3>
    <ul>
      <li>Use theme palette roles and existing component props.</li>
      <li>Start with existing components.</li>
      <li>Keep behaviour and interaction patterns consistent.</li>
      <li>Let the theme handle light and dark mode.</li>
    </ul>
  </div>

  <div className="ds-card">
    <h3>Don’t</h3>
    <ul>
      <li>Hardcode colours or spacing values in product components.</li>
      <li>Create new components without a clear need.</li>
      <li>Override system behaviour without strong justification.</li>
      <li>Add separate light/dark styling inside individual components.</li>
    </ul>
  </div>
</div>

## Light and dark mode

Light and dark mode are supported by default and are handled at the system
level through MUI colour schemes.

Dark-mode status colours are tuned for perceived clarity rather than copied
directly from MUI defaults.

<ul>
  <li>
    Theme switching is controlled via <code>data-mode</code> on the{" "}
    <code>html</code> element.
  </li>
  <li>
    Components should use MUI props, palette roles or theme values so colours
    adapt automatically.
  </li>
  <li>No component should implement separate light/dark logic.</li>
</ul>

<pre>{`<html data-mode="light">`}</pre>
