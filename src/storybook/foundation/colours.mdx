<div>
  <h1>Colour</h1>

  <p>
    DiamondDS uses a layered colour system designed for scientific software.
    Colour is defined once at the primitive level and applied through semantic
    roles so components remain consistent across light and dark themes.
  </p>

  <div className="ds-note">
    <strong>Architecture</strong>
    <div style={{ marginTop: 8 }}>
      <code>Primitives → Semantic tokens → MUI theme mapping → Components</code>
    </div>
    <p style={{ marginTop: 10, marginBottom: 0 }}>
      Theme switching happens via <code>&lt;html data-mode="light|dark"&gt;</code>. Primitives change; semantic
      tokens follow; components inherit automatically.
    </p>
  </div>

  <section className="ds-guidelines">
    <h2>How the layers work</h2>

    <div className="ds-grid">
      <div className="ds-note">
        <h3>Primitives</h3>
        <p>
          Raw colour scales (Radix-style 1–12) for neutrals and intents.
          Light/dark mode is handled here.
        </p>
        <ul>
          <li>Stable: components should not depend on individual steps</li>
          <li>Theme toggle swaps primitive values</li>
        </ul>
      </div>

      <div className="ds-note">
        <h3>Semantic tokens</h3>
        <p>
          Prefer semantic roles over raw primitives. Use tokens like
          <code>--ds-bg-…</code>, <code>--ds-fg-…</code>, <code>--ds-border-…</code>,
          <code>--ds-overlay-…</code>, <code>--ds-intent-…</code>.
        </p>
        <p style={{ marginBottom: 0 }}>
          If a role is missing, add a semantic token — don’t start referencing a primitive step in component code.
        </p>
      </div>

      <div className="ds-note">
        <h3>MUI palette</h3>
        <p>
          MUI components use the palette via props like <code>color="primary"</code>.
          Our theme maps palette slots to intent steps.
        </p>
        <ul>
          <li><code>light → 8</code>, <code>main → 9</code>, <code>dark → 10</code></li>
          <li>Overlays use explicit <code>action. …</code> tokens</li>
        </ul>
      </div>
    </div>

    <div className="ds-note">
      <h3>Rule of thumb</h3>
      <p style={{ marginBottom: 0 }}>
        <strong>Structure</strong> uses background and border roles.
        <strong> Meaning</strong> uses intent tokens.
        <strong> Interaction</strong> uses overlays.
      </p>
    </div>
  </section>

  <h2>Primitives (Radix-style scale)</h2>
  <p>
    Primitives follow a 12-step scale. You don’t usually pick a step in a component;
    primitives exist so semantic roles remain stable while the theme swaps values.
  </p>

  <div className="ds-note">
    <strong>Scale logic</strong>
    <ul style={{ marginTop: 8 }}>
      <li><strong>Steps 1–3</strong>: app backgrounds and base surfaces</li>
      <li><strong>Steps 3–5</strong>: interactive surface states (hover, selected, pressed) — often overlays</li>
      <li><strong>Steps 6–8</strong>: separators + borders (structure → affordance)</li>
      <li><strong>Steps 9–10</strong> solid intent backgrounds (main → hover/active)</li>
      <li><strong>Steps 11–12</strong>: foreground emphasis (intent label/text → maximum contrast)</li>
    </ul>
  </div>

  <h2>Intent text and labels</h2>
  <p>
    A common need is “text that carries meaning”, without turning the whole surface into an intent colour.
    We support this through intent label and intent foreground tokens.
  </p>

  <div className="ds-note">
    <strong>What to use</strong>
    <ul style={{ marginTop: 8 }}>
      <li><code>--ds-intent-…-label</code> for focused labels and active helper text (e.g. focused input label)</li>
      <li><code>--ds-intent-…-subtle-fg</code> for readable intent text on subtle backgrounds (alerts, selected rows)</li>
      <li><code>--ds-intent-…-outlined-fg</code> for links, outlined buttons, chips, and emphasised inline intent</li>
      <li><code>--ds-fg-on-solid</code> for text and icons on solid intent surfaces</li>
    </ul>
  </div>

  <h2>Intent tokens (meaning)</h2>
  <p>
    Intent tokens express meaning. They come in three layers: <code>solid</code>, <code>subtle</code>, and <code>outlined</code>.
    This lets us keep controls calm and readable in data-dense interfaces.
  </p>

  <div className="ds-note">
    <strong>Layers</strong>
    <ul style={{ marginTop: 8 }}>
      <li><code>solid</code> — primary actions and strong emphasis</li>
      <li><code>subtle</code> — contextual surfaces (alerts, selected rows)</li>
      <li><code>outlined</code> — secondary controls and links</li>
    </ul>

  <h2>Using MUI palette vs DiamondDS tokens</h2>
  <p>
    Use MUI palette props for intent (primary, error, success) on MUI components.
    Use DiamondDS semantic tokens for structure, surfaces, borders and overlays — especially in custom components.
  </p>

  <div className="ds-grid">
    <div className="ds-note">
      <strong>Use MUI palette when</strong>
      <ul>
        <li>You’re using MUI components directly</li>
        <li>You want consistent intent behaviour through props</li>
        <li>You’re relying on theme defaults for hover/selected/focus</li>
      </ul>
      <div className="ds-example" style={{ marginTop: 10, marginBottom: 0 }}>

  <h2>Do and don’t</h2>
  <div className="ds-grid">
    <div className="ds-note">
      <strong>Do</strong>
      <ul style={{ marginTop: 8 }}>
        <li>Use semantic tokens for structure (<code>--ds-bg-…</code>, <code>--ds-fg-…</code>, <code>--ds-border-…</code>)</li>
        <li>Use intent tokens for meaning (<code>--ds-intent-…</code> or <code>color="…"</code> on MUI components)</li>
        <li>Use surfaces to separate regions rather than heavy shadows</li>
        <li>Let primitives handle light/dark differences</li>
        <li>Ensure disabled and error states override other styling</li>
      </ul>
    </div>

    <div className="ds-note">
      <strong>Don’t</strong>
      <ul style={{ marginTop: 8 }}>
        <li>Don’t reference primitives directly in components (e.g. <code>--ds-indigo-9</code>)</li>
        <li>Don’t hardcode hex values in component code</li>
        <li>Don’t use colour alone to communicate state (pair with label, icon or layout)</li>
        <li>Don’t mix shadow-based elevation and surface layering unnecessarily</li>
        <li>Don’t invent new colour roles; extend semantic tokens if a role is missing</li>
      </ul>
    </div>
  </div>
