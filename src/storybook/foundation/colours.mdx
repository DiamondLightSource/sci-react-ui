# Colour

<div className="ds-copy">

<p>
  Diamond Design System uses a role-based colour system designed for scientific
  software. Colours are defined through stable UI roles and then mapped into the
  MUI theme, so components remain consistent across light and dark mode.
</p>

</div>

<div className="ds-wide">

<div className="ds-note">
  <strong>Architecture</strong>
  <div style={{ marginTop: 8 }}>
    <code>Role tokens → MUI theme mapping → components</code>
  </div>
  <p style={{ marginTop: 10, marginBottom: 0 }}>
    Theme switching happens via <code>&lt;html data-mode="light|dark"&gt;</code>.
    Role tokens define all colours and change between themes. The MUI theme maps
    those roles into component behaviour, so components update automatically.
  </p>
</div>

</div>

## How the layers work

<div className="ds-wide">

<div className="ds-grid">
  <div className="ds-note">
    <h3>Role tokens</h3>
    <p>
      These are the source of truth for colour in the Diamond Design System. They
      define UI roles directly, such as background, surface, border, text,
      overlay, focus, and intent.
    </p>
    <ul>
      <li>Components should depend on roles, not raw colour values.</li>
      <li>Theme switching swaps role values, not component code.</li>
    </ul>
  </div>

  <div className="ds-note">
    <h3>Theme mapping</h3>
    <p>
      The MUI theme maps those role tokens into palette slots, action states, and
      component defaults.
    </p>
    <p style={{ marginBottom: 0 }}>
      This lets MUI components use familiar props like <code>color="primary"</code>
      while still following Diamond Design System rules.
    </p>
  </div>

  <div className="ds-note">
    <h3>Components</h3>
    <p>
      Components consume those mapped roles. This keeps buttons, fields, tabs,
      chips, and custom UI consistent across themes.
    </p>
    <ul>
      <li>Structure comes from neutral roles.</li>
      <li>Meaning comes from intent roles.</li>
    </ul>
  </div>
</div>

<div className="ds-note">
  <h3>Rule of thumb</h3>
  <p style={{ marginBottom: 0 }}>
    <strong>Structure</strong> uses background and border roles.
    <strong> Meaning</strong> uses intent roles.
    <strong> Interaction</strong> uses overlays and focus tokens.
  </p>
</div>

</div>

## Role tokens

<div className="ds-copy">

<p>
  Role tokens are the colour source of truth in the Diamond Design System. Each
  token represents a specific role in the interface and maps directly to a
  concrete value in light and dark mode.
</p>

</div>

<div className="ds-wide">

<div className="ds-note">
  <strong>Token groups</strong>
  <ul style={{ marginTop: 8 }}>
    <li><strong>Background</strong>: app and page canvas</li>
    <li><strong>Surfaces</strong>: containers and grouped regions</li>
    <li><strong>Borders</strong>: structure and affordance</li>
    <li><strong>Foreground</strong>: text and icon contrast</li>
    <li><strong>Intent</strong>: action, status, and meaning</li>
    <li><strong>Overlays</strong>: hover, selected, focus, disabled</li>
  </ul>
</div>

</div>

## Intent roles

<div className="ds-copy">

<p>
  Intent roles add meaning. They are mapped into the MUI palette and also
  provide container, solid, on-container, and on-solid values for calmer and
  stronger usage patterns.
</p>

</div>

<div className="ds-wide">

<div className="ds-note">
  <strong>How intent works</strong>
  <ul style={{ marginTop: 8 }}>
    <li><strong>main</strong> is the default semantic colour for text, icons, borders, and standard intent usage.</li>
    <li><strong>emphasis</strong> is the stronger interactive or more prominent semantic step.</li>
    <li><strong>accent</strong> supports lighter emphasis and supporting semantic detail.</li>
    <li><strong>container</strong> is a subtle intent surface.</li>
    <li><strong>on container</strong> is the readable foreground for that subtle surface.</li>
    <li><strong>solid</strong> is the stronger filled semantic surface.</li>
    <li><strong>on solid</strong> is the readable foreground for that stronger fill.</li>
  </ul>
</div>

## How roles map to the MUI palette

<div className="ds-copy">

<p>
  The Diamond Design System defines colour through semantic role tokens. The MUI
  palette is a mapping layer that exposes those roles through the fields MUI
  expects, such as <code>light</code>, <code>main</code>, <code>dark</code>, and
  <code>contrastText</code>.
</p>

<p>
  This allows standard MUI components to work with familiar props like
  <code>color="primary"</code>, while Diamond-specific roles such as
  <code>container</code>, <code>on-container</code>, <code>solid</code>, and
  <code>on-solid</code> support richer component behaviour and custom UI.
</p>

</div>

<div className="ds-wide">

<div className="ds-note">
  <strong>Important</strong>
  <p style={{ marginTop: 8, marginBottom: 0 }}>
    MUI palette fields are not the source of truth. They are derived from
    Diamond Design System role tokens so MUI components behave correctly.
  </p>
</div>

</div>

## Surfaces and elevation

<div className="ds-copy">

<p>
  Diamond Design System uses elevation to create hierarchy, but most elevation
  should come from surface colour, borders, and spacing rather than drop shadows.
  This keeps dense scientific interfaces calm, readable, and predictable.
</p>

</div>

<div className="ds-wide">

<div className="ds-note">
  <strong>Surface hierarchy</strong>
  <ul style={{ marginTop: 8 }}>
    <li><code>background</code> → app and page background (<code>--ds-background</code>)</li>
    <li><code>surface</code> → primary container surface (<code>--ds-surface</code>)</li>
    <li><code>surface container</code> → grouped sections (<code>--ds-surface-container</code>)</li>
    <li><code>surface container high</code> → more prominent nested areas (<code>--ds-surface-container-high</code>)</li>
  </ul>
  <p style={{ marginBottom: 0 }}>
    Use drop shadows sparingly, mainly for floating overlays such as menus,
    popovers, dialogs, and tooltips.
  </p>
</div>

<div className="ds-example">
  <p className="ds-kicker">Practical: Paper with surfaces inside</p>
  <pre>{`<Paper elevation={0} sx={{ background: "var(--ds-surface)", border: "1px solid var(--ds-border-subtle)" }}>
  <Box sx={{ background: "var(--ds-surface-container)", p: 2 }}>
    Section title
  </Box>
  <Box sx={{ background: "var(--ds-surface-container-high)", p: 2 }}>
    Interactive area (inputs, controls)
  </Box>
</Paper>`}</pre>
</div>

</div>

## Using MUI palette vs Diamond roles

<div className="ds-copy">

<p>
  Use MUI palette props for intent on MUI components. Use Diamond Design System
  role tokens for structure, surfaces, borders, overlays, and custom UI.
</p>

</div>

<div className="ds-wide">

<div className="ds-grid">
  <div className="ds-note">
    <strong>Use MUI palette when</strong>
    <ul>
      <li>You’re using MUI components directly.</li>
      <li>You want consistent intent behaviour through props.</li>
      <li>You’re relying on theme defaults for hover, selected, and focus behaviour.</li>
    </ul>
    <div className="ds-example" style={{ marginTop: 10, marginBottom: 0 }}>
      <p className="ds-kicker">Example</p>
      <pre>{`<Button color="primary" variant="contained">
  Run scan
</Button>`}</pre>
    </div>
  </div>

  <div className="ds-note">
    <strong>Use Diamond roles when</strong>
    <ul>
      <li>You’re styling layout or containers.</li>
      <li>You’re building a custom component or anatomy.</li>
      <li>You need a specific surface or border role.</li>
      <li>You’re styling non-MUI DOM elements.</li>
    </ul>
    <div className="ds-example" style={{ marginTop: 10, marginBottom: 0 }}>
      <p className="ds-kicker">Example</p>
      <pre>{`<div style={{
  background: "var(--ds-surface-container)",
  border: "1px solid var(--ds-border-subtle)",
  color: "var(--ds-on-surface)",
}}>
  Panel content
</div>`}</pre>
    </div>
  </div>
</div>

</div>

## Do and don’t

<div className="ds-wide">

<div className="ds-grid">
  <div className="ds-note">
    <strong>Do</strong>
    <ul style={{ marginTop: 8 }}>
      <li>Use neutral roles for structure (<code>--ds-background</code>, <code>--ds-surface</code>, <code>--ds-border-*</code>).</li>
      <li>Use intent roles for meaning (<code>--ds-primary</code>, <code>--ds-success</code>, <code>--ds-danger</code>, etc.).</li>
      <li>Use surfaces to separate regions rather than heavy shadows.</li>
      <li>Let the same token names work across light and dark themes.</li>
      <li>Ensure disabled and error states override other styling.</li>
    </ul>
  </div>

  <div className="ds-note">
    <strong>Don’t</strong>
    <ul style={{ marginTop: 8 }}>
      <li>Don’t hardcode hex values in component code.</li>
      <li>Don’t use strong intent colours as general layout backgrounds.</li>
      <li>Don’t use colour alone to communicate state.</li>
      <li>Don’t mix shadow-based elevation and surface layering unnecessarily.</li>
      <li>Don’t invent one-off colour behaviour inside individual components.</li>
    </ul>
  </div>
</div>

</div>
